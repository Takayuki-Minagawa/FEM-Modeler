# AI指示書: 静的Webアプリとして実装するFEM/CAEモデリング支援環境

## 1. 文書の目的

この文書は、**有限要素法(FEM)・熱解析・流体解析向けの前処理支援Webアプリ**を、**静的Webアプリ**として設計・実装するために、AIへ与える指示書である。  
本アプリは、単なるデモではなく、**現状の市販ソフトと同等レベルの操作性・視認性・実務利用性**を目標とする。

本アプリは特に以下を重視する。

- 静的配信で動作すること
- ローカルでデータの入出力ができること
- 入力フォームが使いやすいこと
- 3Dビューワーが実務で使えること
- AIによるモデル生成・補助ができること
- OpenSeesPy / DOLFINx / OpenFOAM へ出力できること

---

## 2. AIへの役割定義

あなたは、以下の役割を兼ねる開発AIである。

- プロダクトオーナー
- UX設計者
- フロントエンド設計者
- 3Dビューワー設計者
- CAE前処理設計者
- データモデル設計者
- エクスポータ設計者
- 検証ルール設計者

あなたは、単にUIを作るのではなく、**実務で破綻しないCAE前処理環境**を設計すること。  
とくに、AIが曖昧な判断を勝手に確定して出力することは避け、**構造化データ・明示的な仮定・検証可能な出力**を常に優先すること。

## 3. 絶対条件

以下は必須条件であり、妥協しないこと。

### 3.1 配布・実行条件

- アプリは**静的Webアプリ**であること
- 配布先は GitHub Pages, Cloudflare Pages, Netlify Static Hosting などを想定できること
- アプリ利用に**専用バックエンドを必須としない**こと
- 主要機能はブラウザ上のみで完結すること
- 必要に応じて WebAssembly, Web Worker, IndexedDB, LocalStorage を利用してよい
- ユーザーデータは基本的にローカルに保持し、明示的な操作なしに外部送信しないこと

### 3.2 製品レベル要件

- 市販のCAE前処理ソフトと比較して、最低限以下で見劣りしないこと
  - モデルの見やすさ
  - 選択のしやすさ
  - 条件設定のしやすさ
  - エラーの分かりやすさ
  - 入出力のしやすさ
  - プロジェクトの再利用性

### 3.3 出力対象

最低限、以下へ出力可能であること。

- OpenSeesPy
- DOLFINx
- OpenFOAM

加えて、内部中間データは必ず**アプリ独自JSON形式**で保存・再読込可能とすること。

## 4. 開発の基本方針

### 4.1 直接ソルバ記述を主データにしない

AIは、OpenSeesPy / DOLFINx / OpenFOAM の個別書式を直接の正本データにしてはならない。  
まず、以下の情報を持つ**共通中間表現(IR: Intermediate Representation)**を設計すること。

- プロジェクト情報
- 単位系
- 形状情報
- トポロジ名・名前付き選択集合
- 材料
- 断面・物性
- メッシュ方針
- 境界条件
- 荷重条件
- 初期条件
- 解析種別
- ソルバ固有オプション
- 検証結果
- エクスポート設定

すべての入出力は、原則としてこの共通中間表現を起点に行うこと。

### 4.2 AIの役割は「自由生成」ではなく「構造化支援」

AIは以下を行ってよい。

- 自然言語から構造化入力候補を作る
- フォームの既定値を提案する
- 名前付き面・線・体積を提案する
- メッシュ細分領域候補を提案する
- 材料候補を提案する
- 境界条件候補を提案する
- 出力形式別の不足項目を指摘する
- エラー原因の候補を説明する

AIは以下を無断で行ってはならない。

- 単位系の勝手な確定
- 接触条件の勝手な確定
- 荷重方向の勝手な反転
- 材料物性の未確認な自動採用
- 解析妥当性の断定
- 出力ファイルの正当性保証の断定

### 4.3 名前付き選択を必須化する

実務上の再利用性を確保するため、以下を基本原則とする。

- 面・辺・頂点・体積・部材には可能な限り意味名を持たせる
- 例: `fixed_face`, `inlet`, `outlet`, `bolt_hole_1`, `symmetry_x`, `heat_source`, `frame_col_01`
- UI上で名前付き選択を一覧・編集・色分けできること
- 各ソルバ出力では、この名前付き選択がタグ・group・patch・marker に対応すること

## 7. UI/UXの最重要方針

### 7.1 フォームは「分かりやすさ最優先」

フォームは、市販ソフトと比較して以下を満たすこと。

- 単位が常に明示される
- 必須項目と任意項目が区別される
- 入力順序が自然である
- 選択対象とフォームが同期する
- 3D上で選んだ対象がどのフォームに対応するか明確である
- 初学者向け説明と実務者向け簡潔表示を切り替えられる

### 7.2 多段フォーム構成

各設定は以下の流れで統一すること。

1. 対象選択
2. 名前確認
3. 条件種別選択
4. 値入力
5. 単位確認
6. 適用プレビュー
7. 検証結果表示

### 7.3 フォーム種類

以下のUI部品を使い分けること。

- 数値入力
- 単位付き入力
- ベクトル入力
- テーブル入力
- 一括貼付入力
- ドロップダウン
- 検索付きセレクト
- トグル
- ラジオボタン
- ステッパー
- 色付きタグ選択
- 条件カード一覧

### 7.4 操作性

必須操作。

- クリック選択
- 範囲選択
- Ctrl/Shift の追加選択
- 非表示/単独表示
- 選択の反転
- 名前付き選択へ保存
- 3D選択とリスト選択の双方向同期

## 14. AI支援の仕様

### 14.1 AI支援の目的

AIは、ユーザーの代わりに勝手に決める存在ではなく、以下を行う支援機能である。

- 曖昧な入力を構造化する
- 抜け漏れを発見する
- 典型設定を提案する
- ソルバ出力差異を埋める
- エラー原因を言語化する

### 14.2 AIの基本動作

AIは常に以下の形式で応答を組み立てること。

1. 認識した内容
2. 仮定した内容
3. 不足している内容
4. 推奨操作
5. 生成または修正する構造化データ

### 14.3 AIが優先すべき入力情報

優先順位は以下とする。

1. ユーザーがフォームで明示した値
2. ユーザーが選択した対象
3. 既存プロジェクト内の名前付き選択
4. テンプレート既定値
5. AI推定値

### 14.4 AIの禁止事項

- 確証のない物理条件の断定
- 解析の成立保証
- エクスポート後の完全動作保証
- 入力ミスの隠蔽
- 重大な不足項目の無視

### 14.5 AIが提案すべき内容の例

- 「この面名は `inlet` と推定されます」
- 「拘束不足の可能性があります」
- 「OpenFOAM 出力には乱流モデル指定が不足しています」
- 「DOLFINx 出力には facet tag が必要です」
- 「OpenSeesPy 出力には断面情報が不足しています」

## 17. 推奨アーキテクチャ方針

実装は以下の層に分けること。

1. UI層
2. アプリ状態管理層
3. 幾何・選択・表示層
4. 共通中間表現(IR)層
5. 検証ルール層
6. AI支援層
7. エクスポート層

### 17.1 状態管理

- 単一の信頼できる状態ソースを持つこと

### 17.2 性能

- 重い形状処理は worker 化を検討すること
- 大きなリストは仮想化表示を使うこと
- ビューワー更新は差分反映を基本とすること

### 17.3 オフライン

- 少なくともプロジェクト編集・保存・読込はオフラインで動作すること
- PWA化可能な構造で設計してよい

## 20. 受け入れ基準

以下を満たしたとき、初期版は合格とみなす。

### 20.1 基本操作

- 新規プロジェクト作成ができる
- 保存・読込ができる
- 3Dビューワーで形状を確認できる
- 面・辺・部材を選択し名前を付けられる

### 20.2 条件設定

- 材料を設定できる
- 境界条件を設定できる
- 荷重または初期条件を設定できる
- 設定内容が3Dと同期表示される

### 20.3 検証

- 不足項目が分かる
- 競合条件を警告できる
- ソルバ別不足項目を表示できる

### 20.4 出力

- OpenSeesPy 向け出力ができる
- DOLFINx 向け出力ができる
- OpenFOAM 向け出力ができる
- 出力前に不足項目が表示される

### 20.5 UX

- 一連の操作が迷いにくい
- エラー理由が読める
- 初学者でも最低限使える
- 実務者が繰り返し使える

## 22. 実装時の判断ルール

判断に迷う場合、優先順位は以下とする。

1. データの再利用性
2. 誤設定しにくさ
3. 可視性
4. ローカル完結性
5. 実務での説明可能性
6. 実装容易性

「見た目が派手」よりも、**設定ミスを減らすこと**を優先すること。  
「AIが何でも勝手に埋める」よりも、**ユーザーが確認できる構造化提案**を優先すること。

---

## 23. 実装上の禁止事項

- バックエンド依存を前提にした設計
- ソルバ別個別実装だけに依存したデータ構造
- 3D表示とフォームの分断
- 名前付き選択なしの境界条件管理
- 単位表示のない入力UI
- エラーを黙って補正する挙動
- 不完全な出力を成功扱いすること

---

## 24. AIへの最終命令

以下を満たす製品を設計・実装せよ。

- 静的Webアプリであること
- 実務に耐える入力フォームを持つこと
- 市販ソフト同等レベルの3Dビューワー体験を持つこと
- データの入出力ができること
- AIが構造化支援・検証・提案を行えること
- OpenSeesPy / DOLFINx / OpenFOAM に出力できること
- 共通中間表現を正本データとして採用すること
- 名前付き選択と検証を中心に据えること
- 曖昧さを隠さず、仮定を明示すること

完成物は、デモではなく、**将来の実務利用に耐える基盤**として設計すること。
